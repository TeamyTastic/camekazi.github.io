AI Collaboration
Session Designer

Design collaborative AI sessions for maintenance tasks and unfamiliar codebases. A lightweight starting point, not a mandate. Adapt freely.

What this is: A play — a tactical, research-backed exercise designed to make it easier to solve a meaningful challenge: how do teams learn to work effectively with AI?

It includes detailed instructions to help develop new capabilities and ends with actionable items to accelerate progress. Not a methodology to follow — an experiment to adapt. As Adaptavist teams run sessions, we'll share learnings here.

Get started in 3 steps

Est. time: ~5 minutes to set up, 1–3 hours for a first session.

Step 1: Gather 2–3 people

Assign three roles to start:

RoleWhat they doKey rule
DriverTypes prompts, executes code. Rotates every 15 minutes.Does NOT decide what to do — only implements
NavigatorDirects the Driver at intent level: "Add a test for null input"Describes what, not how — AI fills the gap
Red TeamChallenges AI output. "What could go wrong? What did it miss?"The AI is wrong until proven right

More roles exist — see the Roles tab. Start with these three.

Step 2: Set a 3-hour time limit

Driver rotates every 15 minutes (use the Timer tab). Three hours is long enough to make real progress, short enough to stay energized.

If 3 hours feels too much, start with 90 minutes. The key constraint is: commit to finishing, not to a time length.

Step 3: Pick a real task

Best starting points:

  • A maintenance bug in an app the team hasn't touched recently
  • An unfamiliar codebase the team needs to learn
  • A code review where AI can explain what the code does

See the Scenarios tab for detailed starting points.

Materials required

  • AI tool access (Claude, Copilot, Cursor, etc.)
  • Shared screen (projector, screenshare, or single laptop)
  • Timer (use the Timer tab or a phone)
  • Optional: recording for transcript analysis afterwards

Beliefs to unlearn

These might feel uncomfortable — that's the point. The discomfort is the learning.

Old beliefNew reality
"Code must be hand-written"AI generates, humans curate and validate
"Every line needs manual review"Specialized review agents catch categories of issues; humans review judgment calls
"First attempts should be good"~95% garbage rate is normal — the speed of iteration matters, not first-draft quality

Value

Why collaborative AI sessions create compounding business value — not just better code.

Sources of value

Revenue diversification

Teams can maintain more apps simultaneously (4→8+), increasing addressable ARR. Shared context means switching between codebases costs minutes, not days. More apps in play = more revenue streams protected.

Accelerated onboarding

New team members learn codebases faster through collaborative sessions than solo study. AI fills context gaps in real time, and the group provides immediate answers to "why was it built this way?" questions that documentation never captures.

Reduced key-person risk

Knowledge trapped in one person's head becomes shared team knowledge. Bus factor goes from 1 to N. Every session is a live knowledge transfer — no separate documentation effort required.

Higher quality through diverse review

Multiple perspectives plus AI critique catches issues a solo developer misses. The Red Team role makes this structural, not accidental — someone is always asking "what could go wrong?"

AI literacy by doing

The team builds AI collaboration skills through real work, not separate training. Satisfies Gartner's recommendation to embed AI learning into daily workflows rather than running it as a standalone programme.

Psychological safety for learning

The group format normalises not-knowing. "We're all figuring this out together" removes the stigma of asking questions. People experiment with AI prompting when the cost of a bad prompt is shared, not personal.

Compound knowledge

Each session's learnings — transcript analysis, documented patterns, updated AI context templates — make the next session better. The system improves itself. Teams that run 10 sessions are dramatically better than teams that run 1.

Something is broken but nobody knows why. The team needs to investigate together.

Reproduce issue

→

First prompt: "Reproduce this bug first—don’t fix it. Gather all logs and info. What’s the root cause?"

Real example: Rate limits were being swallowed silently. The job hit the limit, failed, and never resumed. Solution needed job resumption + retry logic, not just error handling.

Unfamiliar legacy codebase

The team needs to work in code nobody currently understands. AI becomes the tour guide.

AI explains architecture
→
Identify relevant files
→
Make small test change

First prompt: "Research how others solved this. Find: official docs, common migration issues, edge cases from real implementations."

Real example: “Upgrade path from version X to Y” → Found official guide + 3 blog posts from engineers who hit the same edge cases. Three minutes of research prevented hours of trial-and-error.

New feature in unfamiliar area

You need to add something to a part of the codebase nobody on the team wrote. Scope may be unclear.

Analyse what exists
→
Check system limits
→
Compare 3 approaches
→
Implement best fit

First prompt: "Before coding, analyse: (1) what we’ve done before, (2) system/API limits, (3) three possible approaches with tradeoffs."

Real example: “Bulk archive 53,000 emails” looked easy. Research revealed rate limits + system timeouts = 3-day architectural challenge avoided upfront.

Code review / understanding

The goal is comprehension, not shipping. AI generates documentation the team validates.

AI generates docs
→
Team validates
→
Improve clarity

First prompt: "Document this module as if for a new team member. Include: purpose, key functions, data flow, and any gotchas you spot."

Code review quality

AI acts as a specialist reviewer, checking dimensions humans often miss under time pressure.

Define review criteria
→
AI reviews against each
→
Team discusses findings

First prompt: "Review this PR against: security, performance, maintainability, test coverage. Flag patterns that violate our standards."

Real example: Added to review checklist: “For background jobs calling external APIs—does it handle rate limits? Does it retry? Does it avoid partial states?”

These are starting points, not scripts. Each team's context is different — adapt the prompts and flow to what actually works for you.

Roles

Start with 3 roles. Add others as you gain experience. Minimum 2 humans — AI augments, doesn't replace the social learning.

Driver

Level 1
Participant

Level 2 — Support

Level 2
Researcher

Level 2
Sponsor
Amplify underheard perspectives. Support least-privileged contributors. Recognise moments of excellence.
Level 2
Rear Admiral
Offer Navigator guidance privately. Provide minimal cues. Guide at highest abstraction level.

Level 3 — Quality

Level 3
Automationist
Identify repeated tool tasks. Notice repeated processes. Recognise boilerplate. Suggest automation.
Level 3
Nose
Flag long lines, complex conditionals, duplication. Name unnamed things. Propose improvements.
Level 3
Archivist
Record alternatives visibly. Document ideas on whiteboard. Display current task. Capture decisions.

Level 4 — Culture

Level 4
Conductor
Combine ideas (discard the first). Elicit quiet members. Make ridiculous proposals. Defer to junior voices.
Level 4
Traffic Cop
Propose new agreements. Alert to process violations. Support adherence. Identify outdated agreements.
Level 4

Express appreciation (SBI model). Share joy. Acknowledge happiness. Retrospect positive moments.
Level 4
Major Pain
Articulate difficulties. Demonstrate empathy. Surface ignored pain points. Call retrospectives on pain.

Session Setup

Practical setup — physical, remote, cadence, ground rules. Est. time: 20 minutes first time, 5 minutes after.

Before you start: onboard your AI

Set the AI up for success at session start. Paste relevant context so it follows your team's conventions from the first prompt.

AI context checklist

ContextWhat to sharePriority
Coding standardsDocument your team's patterns (style guide, naming conventions, approved libraries, deprecated patterns) so AI follows your conventions, not generic onesMust
Git workflowBe explicit: branching model, commit message format, what triggers a PR reviewMust
Communication clarityWrite detailed specs. The more specific, the better the output. Ambiguity = mediocre resultsMust
Testing requirementsState expected coverage, test patterns, failure modes you care aboutShould
Debugging protocol“Reproduce before fixing”—don’t guess. Paste logs, error messages, or bug threads firstShould
Iteration protocolAfter corrections, document what worked in your team’s AI guidelines so patterns compound across projectsShould
Tech stack constraintsApproved libraries, deprecated patterns, version requirementsNice
Documentation expectationsInline comments, README updates, ADR formatNice

Example opening prompt: "We follow these standards: [paste]. Please adhere throughout this session and flag any deviations."

Session recording & transcript analysis (if opted in)

If the team has a Session Recorder (see Roles tab), record your session (audio or screen) and analyse the transcript with AI afterwards. This is how the team learns about how they collaborate, not just what they built.

After the session: reflection prompts

Use these for manual discussion or paste your transcript into AI and ask it to answer them:

#QuestionWhat it reveals
1Did everyone contribute roughly equally?Participation balance, dominance patterns
2Where did we get stuck? What unblocked us?Bottlenecks and effective interventions
3What assumptions did the AI make that we had to correct?Context gaps in AI onboarding
4What patterns emerged in how we worked together?Collaboration dynamics, role effectiveness
5What would we do differently next time?Specific improvements to try

AI analysis prompt: "Here's a transcript of our collaborative coding session. Analyse it against these 5 questions and give us specific, actionable feedback. Be honest about what didn't work."

Ground rules & psychological safety

Ground rules aren't bureaucracy — they're the safety infrastructure that makes real challenge possible. Without them, people self-censor. With them, people speak up.

The foundation: Kindness, Consideration, Respect

Woody Zuill's core principle. Sounds obvious — but explicitly stating it changes behaviour. People who aren't naturally inclined to these quickly improve when the whole group commits.

ValueIn practiceObservable signal
KindnessConcern for your group — check in, notice when someone goes quietPeople name difficulty without being asked
RespectDisagree with ideas, never attack people"I see it differently" not "that's wrong"
ConsiderationActively listen — let ideas finish before respondingPeople build on each other's suggestions

Working agreements

AgreementWhy it builds safety
"Yes, and…"Build on what came before. No deleting and starting over when you take the keyboard.
Strong-style navigation"For an idea to go from your head to the computer, it must go through someone else's hands." Decouples ego from execution.
No devicesFull attention = full belonging. Announce before session so people don't bring laptops.
Debate in code, not in the abstractTwo approaches? Try both. Data decides, not hierarchy. Removes "who's right" pressure.
Chatham House RuleWhat's said in the session stays in the session. Mistakes are learning, not ammunition.
Deliberateness over impulsiveness"Read the AI output together before we hit accept." Prevents reflexive shipping.

AI-specific safety additions

AI introduces new psychological risks — people feel exposed when they don't know how to prompt, or embarrassed when AI produces something better than they could.

AgreementThe risk it addresses
"No such thing as a bad prompt"Prompt shame — people won't experiment if they fear looking foolish
Read output aloud togetherPrevents one person silently deciding if AI output is "good enough"
Everyone gets a turn promptingAI-confident people dominate; novices never learn the feel of it
"The AI is wrong until proven right"Removes authority bias — nobody defers to AI just because it sounds confident
Name what you don't knowAI makes knowledge gaps visible — normalise saying "I don't understand this output"

Facilitator: calibrating safety in real-time

Safety isn't a checkbox — it's a live reading. Watch for these signals and adjust.

Shutdown signalFacilitator response
Silence after rotation — new driver freezesNavigator: use Intention → Location → Details ("We want to add a test. Open the test file. Type…")
"Yes, but…" responses increasingName it: "I notice we're in 'yes but' mode — let's try the idea for 5 minutes and then evaluate"
One person hasn't spoken for 10+ minutes"[Name], what are you seeing that we might be missing?"
Defensive body language after AI critique"Remember: we're debugging the prompt, not the person who wrote it"
Someone apologises for a "bad" prompt"That prompt taught us what the AI needs — that's exactly how this works"
Group defers to the most senior personRotate so junior person navigates next; enforce strong-style strictly
"Truth without safety creates trauma, not transformation. Safety doesn't eliminate discomfort — it enables it."Psychological Safety Calibration principle

For deeper assessment: rate the group on Attitudes (openness to challenge), Behaviours (do actions match stated values?), and Context (does the org reward truth-telling or peace-keeping?) on a 1–5 scale. Average above 3.5 = tread carefully with challenge intensity.

Kamer's 8 tips for experiments

#TipApplied to collaborative AI sessions
1The test comes from you — genuine passion, not assignedChampion volunteers, don't conscript
2Define the desired effect — concrete, measurable"Resolve first ticket in <2 hours"
3Be bold — radical thing, non-radical scaleFull collaboration week, not a timid 1-hour trial
4One thing at a time — change one variableCollaborative sessions OR new AI tool. Not both simultaneously
5Right time horizon — 8 weeks ideal, 16 max2–3 sprint cycles to build real skill
6Don't kill it early — persist past discomfort3 sessions is too few. Commit to 8+
7Retrospect regularly — monthly learningsEnd-of-session retro + monthly pattern review
8Expect frustration — 2 steps forward, 1 back"Why is this slower?" is normal at first

The T-shape model

From Rodney Evans at The Ready: teach one universal practice across all teams (the bar of the T — e.g. retrospectives). Let a smaller adventurous group experiment widely (the stem of the T). That pioneering group becomes the organizational model.

"Start with an adventurous group already doing things differently — not a group stuck in dysfunction. It will be more inspiring."Jurriaan Kamer, The Ready

What's next: after your first sessions

This isn't one-and-done. Each session should make the next one better.

Close the feedback loop

#StepHow
1Review transcript analysis togetherUse the reflection prompts from the Setup tab. Do this as a team, not solo.
2Document what worked / what to changeKeep a shared "session log" — even a simple list in a wiki page.
3Update team practicesWhat AI context template needs updating? What rotation length felt right? What roles helped?
4Share learnings with other teamsOptional but powerful. Your first sessions become case studies for others.

Questions to evolve your practice

  • What patterns emerged across sessions? What would we systematise?
  • Are we getting better at prompting? At challenging AI output?
  • Is the team's shared understanding of the codebase actually growing?
  • Which roles added the most value? Which felt unnecessary?

Research & Evidence

Supporting research for those who want to understand the "why" behind these practices. Optional reading — not required to get started.

Enabling constraints

John Cutler: "The thoughtful design and deployment of constraints is a superpower." Every practice here is a deliberately designed boundary that drives creativity rather than restricting it.

ConstraintEnabling intentRisk if misapplied
Rotation timerForces knowledge transfer through handover discomfortBecomes performative — rotating without actually sharing context
Strong-style"Ideas must go through someone else's hands" — prevents solo hero modeDriver becomes passive typist; Navigator micromanages keystrokes
Single computerEliminates multitasking; forces shared attentionBoredom/disengagement when participant role feels passive
No backlogs of ideasPresent-moment focus; decide what matters NOWValuable observations lost because no capture mechanism
8-week experiment horizonLong enough to build skill, short enough to evaluateKilled at week 2 because "it feels slower" (see Kamer Tip 6)
"Same constraint, dramatically different results. Context determines whether a deadline is a creative forcing function or a performative metric."John Cutler, TBM 386

Cutler's 4 signs of well-designed constraints: Clarity (easy to understand), Tension (requires ongoing judgment), Full-loop (connects back to learning), Active engagement (not bureaucratic hoops).

Warning stickers

At session start, ask each participant: "What's your known risk in this kind of session?" Public self-declaration builds vulnerability and gives the group permission to name patterns when they appear.

RiskAntidoteWhy it matters for safety
Analysis paralysis"We decide in 5 minutes"Perfectionism shuts others out — speed creates space
Impatience / wants to build"Let's read the output together"Rushing skips shared understanding
Dominates keyboardTimer enforces driver swapStructure protects what social pressure can't
Goes quiet / defers"What are you seeing?"Silence ≠ agreement; draw out dissent gently
Prompt perfectionism"Just try it — we'll iterate"Fear of "bad" prompts prevents learning
AI authority bias"What would we do without the AI?"Group must maintain independent judgment

Facilitator tip: Emotions in collaborative sessions are data, not problems. Quick read — Happy (needs met), Frustrated (blocked), Bored (not contributing enough), Anxious (exposed), Confused (pace too fast). Name the emotion, address the need.

The AI literacy case

Gartner (Carlton Sapp, 2025): enterprises need Human-AI Collaboration Curricula with 6 domains. Collaborative AI sessions deliver all 6 by design.

Gartner domainHow collaborative sessions deliver it
Workflow integrationLearning embedded in real work — not a separate training day
Collaborative outcomesTickets resolved, ramp-up time, shared understanding — all measurable
Supervision & observabilityWhole team watches AI output — humans-in-the-loop by default
Knowledge expansionEvery rotation transfers AI prompting patterns between people
Role-based upskillingStructured roles and levels = progression paths
Executive AI governanceRed Team role explicitly critiques AI output; deliberateness over impulsiveness
"Human-AI collaboration curricula must be deployed the same way cybersecurity awareness training is — as a shared, measurable competency tied to business outcomes."Gartner, Build Human-AI Collaboration Curricula (2025)

Gartner recommends tiered learning journeys (multiple paths to same standard), hands-on training (hackathons, AI labs), and embedding learning into daily workflows. Collaborative AI sessions are all three at once.

Evidence from the field

Crisp (Sweden) — Daily 3hr sessions

AI as "superintelligent advisor — always available, never tired." Eliminated code reviews, removed key-person dependencies, reduced technical debt.

"We tried splitting up to hit a deadline, but reintegration was painful. We're faster together."

Atlassian — Define → Spec → Generate → Review → Iterate

2–4 developers with rotating facilitator. Features in 1–2 days vs typical week. AI draft accuracy: ~50% → ~80% by project end.

Pilot metrics: Blocker resolution +39pp, Innovation exploration +26pp, Collaboration anxiety 52% → 0%.
