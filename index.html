Claude Code Learning System

Compounding Learning System Built on one question: "Did the system learn, or did I just get through today?" CLI
Agent Self-Improvement AI agents that propose and track their own improvements — writes proposals to staging, humans approve via review workflow Claude Code

Philosophy: Every interaction should make future interactions better. Capture what worked AND failed, grep when needed, delete what's stale.

How it works:

  • Friction detection — /hookify analyzes conversations for corrections and frustration, then proposes prevention rules automatically
  • Pattern promotion — Observations that recur 3+ times get promoted to rules, skills, or agents
  • Agent self-improvement — Domain agents propose their own improvements based on session patterns; humans approve via /review-improvements
  • Coached-thinker mode — Nudges toward action over analysis paralysis; asks "What's the smallest step?" not "Have you considered..."
  • Domain agents — Immunity to Change coaching (Kegan), systems thinking (Meadows), organizational sensemaking (Cynefin + Estuarine)
  • 68+ skills — JTBD, ONA network analysis, Wardley Maps, financial modeling, stakeholder analysis, and more

Two files. Simple format. Grep to search. Built for individuals and teams who want to systematize expertise instead of rediscovering it.

Insights from Using AI Agents

Patterns from building 68+ skills and 12 agents with Claude Code

The Learning Loop

  1. Failures are gold — Document what failed, not just what worked. The ❌ field is the most valuable part of any learning.
  2. 5+ occurrences, then operationalize — Wait for patterns to prove themselves before building infrastructure around them.
  3. Propose, don't auto-apply — AI should suggest changes; humans approve. Prevents drift and maintains trust.
  4. Grep when needed, delete what's stale — One searchable file beats elaborate tracking systems.

Agent Design

  1. Match complexity to artifact — Simple → Command. Complicated → Skill. Complex → Agent. Wrong artifact type = wasted effort.
  2. Extend, don't create — Adding 200 lines to existing code beats 500 lines standalone. Check what exists first.
  3. Skill names should be grep-able — "subtext-analysis" beats "hermeneutic-observation". Name for discovery.
  4. Features are prompts, tools are primitives — Define outcomes in natural language. Tools provide capability, not behavior.
  5. Agent-user parity — Whatever the user can do, the agent can do. Whatever the user can see, the agent can see.
  6. Don't LLM what regex can do — Use deterministic handling for structured inputs. Save the LLM for genuinely ambiguous cases.

Working with AI

  1. Brainstorm before simplifying — Present 5-7 options before rewriting. Let humans shape direction.
  2. URL-driven task initiation — Concrete URLs beat abstract descriptions. "Install X from [link]" works; "add that library" doesn't.
  3. After 3 failed fixes, disable and move on — Know when to stop debugging. Revisit with fresh context later.
  4. Inject domain context into prompts — "DO NOT suggest X for these items" prevents obvious mistakes. LLMs need explicit exclusions.
  5. Check native before customizing — Query the changelog first. The feature you're building might already exist.
