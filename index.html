Claude Code Learning System

Compounding Learning System Built on one question: "Did the system learn, or did I just get through today?" CLI

Philosophy: Every interaction should make future interactions better. Capture what worked AND failed, grep when needed, delete what's stale.

How it works:

  • Friction detection ‚Äî /hookify analyzes conversations for corrections and frustration, then proposes prevention rules automatically
  • Pattern promotion ‚Äî Observations that recur 3+ times get promoted to rules, skills, or agents
  • Agent self-improvement ‚Äî Domain agents propose their own improvements based on session patterns; humans approve via /review-improvements
  • Coached-thinker mode ‚Äî Nudges toward action over analysis paralysis; asks "What's the smallest step?" not "Have you considered..."
  • Domain agents ‚Äî Immunity to Change coaching (Kegan), systems thinking (Meadows), organizational sensemaking (Cynefin + Estuarine)
  • 75 skills ‚Äî JTBD, ONA network analysis, Wardley Maps, financial modeling, stakeholder analysis, power dynamics, and more

Two files. Simple format. Grep to search. Built for individuals and teams who want to systematize expertise instead of rediscovering it.

Insights from Using AI Agents

Patterns from building 75 skills and 7 specialized agents with Claude Code

The Learning Loop

  1. Failures are gold ‚Äî Document what failed, not just what worked. The ‚ùå field is the most valuable part of any learning.
  2. 5+ occurrences, then operationalize ‚Äî Wait for patterns to prove themselves before building infrastructure around them.
  3. Propose, don't auto-apply ‚Äî AI should suggest changes; humans approve. Prevents drift and maintains trust.
  4. Grep when needed, delete what's stale ‚Äî One searchable file beats elaborate tracking systems.
  5. Evidence before infrastructure ‚Äî Analyzed 20 PRs ‚Üí only 3 had critical issues ‚Üí built minimal tooling. Check the data before building the system.

Agent Design

  1. Match complexity to artifact ‚Äî Simple ‚Üí Command. Complicated ‚Üí Skill. Complex ‚Üí Agent. Wrong artifact type = wasted effort.
  2. Extend, don't create ‚Äî Adding 200 lines to existing code beats 500 lines standalone. Check what exists first.
  3. Skill names should be grep-able ‚Äî "subtext-analysis" beats "hermeneutic-observation". Name for discovery.
  4. Features are prompts, tools are primitives ‚Äî Define outcomes in natural language. Tools provide capability, not behavior.
  5. Agent-user parity ‚Äî Whatever the user can do, the agent can do. Whatever the user can see, the agent can see.
  6. Don't LLM what regex can do ‚Äî Use deterministic handling for structured inputs. Save the LLM for genuinely ambiguous cases.
  7. Multi-agent review cuts scope ‚Äî 3 agents reviewing a plan identified 60-70% cuts. 45 min plan ‚Üí 12 min implementation.

Working with AI

  1. Brainstorm before simplifying ‚Äî Present 5-7 options before rewriting. Let humans shape direction.
  2. URL-driven task initiation ‚Äî Concrete URLs beat abstract descriptions. "Install X from [link]" works; "add that library" doesn't.
  3. After 3 failed fixes, disable and move on ‚Äî Know when to stop debugging. Revisit with fresh context later.
  4. Inject domain context into prompts ‚Äî "DO NOT suggest X for these items" prevents obvious mistakes. LLMs need explicit exclusions.
  5. Check native before customizing ‚Äî Query the changelog first. The feature you're building might already exist.
  6. Match planning fidelity to task size ‚Äî One-line fix? Skip planning. Multi-file clear scope? Light plan. Major feature? Full ceremony with parallel agents.
  7. CLI over MCP for reliability ‚Äî CLI tools are predictable and debuggable. MCP can miss data. When reliability matters, use the boring option.
